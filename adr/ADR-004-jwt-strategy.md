# ADR-004: JWT 署名アルゴリズムとトークン戦略

**ステータス:** 採用済み（Phase 2: HS256、Phase 5+: RS256 へ移行予定）  
**作成日:** 2026-05-24  
**作成者:** Dao Nguyen

---

## コンテキスト

JWT 認証を実装するにあたり、以下の設計判断が必要だった:

1. **署名アルゴリズム**: HS256（共有秘密鍵）vs RS256（RSA 非対称鍵）
2. **Access Token の有効期限**: セキュリティと UX のバランス
3. **Refresh Token の管理方式**: ステートレス vs ステートフル（DB 管理）
4. **Refresh Token のローテーション**: 固定 vs ローテーション

---

## 決定事項

### Phase 2（現在）

| 項目 | 決定内容 |
|---|---|
| 署名アルゴリズム | **HS256**（HMAC-SHA256、共有秘密鍵） |
| Access Token 有効期限 | **15 分** |
| Refresh Token 有効期限 | **7 日** |
| Refresh Token 管理 | **DB 管理**（SHA-256 ハッシュ値を `refresh_tokens` テーブルに保存） |
| Refresh Token ローテーション | **採用**（リフレッシュのたびに新しいトークンに差し替え） |

### Phase 5+（将来移行）

| 項目 | 変更内容 |
|---|---|
| 署名アルゴリズム | HS256 → **RS256**（RSA 非対称鍵） |
| 移行理由 | 将来のマイクロサービス化・別サービスへの JWT 共有 |

---

## 理由

### HS256 を Phase 2 で採用する理由

| 観点 | HS256 | RS256 |
|---|---|---|
| 実装の複雑さ | シンプル（秘密鍵1つ） | 複雑（公開鍵・秘密鍵のペア管理） |
| 鍵管理 | 環境変数1つ | キーペアの生成・配布が必要 |
| 検証速度 | 速い | HS256 より若干遅い |
| マルチサービス対応 | 秘密鍵の共有が必要（セキュリティリスク） | 公開鍵の配布のみで検証可能 |
| 現在のアーキテクチャ | シングルサービス → HS256 で十分 | - |

現在はモジュラーモノリスの単一サービスであり、HS256 で必要十分。鍵管理のシンプルさを優先する。

### Access Token を 15 分に設定する理由

- 短すぎると UX が悪化（頻繁なリフレッシュ）
- 長すぎるとトークン漏洩時のリスクが増大
- 15 分は業界標準的な値であり、Refresh Token によるサイレントリフレッシュで UX を維持できる

### Refresh Token を DB 管理にする理由

ステートレス（自己完結型 JWT）にすると:
- ログアウト後もトークンが有効になる（無効化できない）
- 盗まれたトークンを失効させられない

DB 管理にすることで:
- ログアウト時に即座に無効化できる
- セキュリティインシデント時に特定ユーザーの全トークンを無効化できる

### Refresh Token ローテーションを採用する理由

リフレッシュのたびに新しい Refresh Token に差し替えることで:
- 盗まれた Refresh Token を使用された際に検知できる（同じトークンが2回使用された場合は不正アクセス）
- 定期的にトークンが更新され、セキュリティが向上する

## 実装

### JWT ペイロード

```json
{
  "sub": "<user_id (UUID)>",
  "role": "ROLE_SELLER",
  "iat": 1748080800,
  "exp": 1748081700
}
```

ユーザー詳細情報（名前・メール等）は含めない。必要な場合は `/users/me` API で取得する。

### Refresh Token の保存

```sql
-- DB にはトークンのハッシュ値のみ保存（平文は保存しない）
INSERT INTO refresh_tokens (user_id, token_hash, expires_at)
VALUES (?, SHA256(rawToken), NOW() + INTERVAL '7 days');
```

### ローテーション失敗の検知と対応

```
1. クライアントが revoked な Refresh Token でリフレッシュを試みる
   → トークンが既に revoked = TRUE または存在しない
   → 401 TOKEN_INVALID を返す
   → 念のため同一ユーザーの全 Refresh Token を失効（再ログインを要求）
```

## 結果

### 得られるもの

- シンプルな鍵管理（環境変数1つ）
- ログアウト・盗難時の即座の無効化
- Refresh Token の定期更新によるセキュリティ向上

### 受け入れるトレードオフ

- Refresh Token のリフレッシュのたびに DB アクセスが発生（ステートレスより重い）
- 将来 RS256 に移行する際に鍵のローテーション作業が必要

### Phase 5+ での RS256 移行パス

```
1. RSA キーペアを生成（2048bit 以上）
2. 秘密鍵を環境変数またはシークレットマネージャーに設定
3. 公開鍵を JWKS エンドポイントで公開（/.well-known/jwks.json）
4. JwtTokenProvider の署名ロジックを HS256 → RS256 に変更
5. 既存の HS256 トークンが期限切れになるまで互換性を維持
```
