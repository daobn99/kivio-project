# ADR-001: モジュラーモノリスの採用

**ステータス:** 採用済み  
**作成日:** 2026-05-24  
**作成者:** Dao Nguyen

---

## コンテキスト

Kivio はマルチベンダー型マーケットプレイスであり、以下の複数のドメインを持つ:

- 認証・ユーザー管理（identity）
- ショップ・商品・カテゴリー（catalog）
- カート・注文・決済（order）
- リアルタイムチャット（messaging）
- 通知・レビュー・プラットフォーム設定（notification, review, platform）
- 監査ログ（audit）

アーキテクチャとして **マイクロサービス** と **モジュラーモノリス** の2つの選択肢があった。

---

## 決定事項

**モジュラーモノリス**（単一デプロイ単位、内部はドメインパッケージで明確に分離）を採用する。

---

## 理由

### モジュラーモノリスを選んだ根拠

| 観点 | モノリス | マイクロサービス |
|---|---|---|
| 開発速度 | 速い（単一コードベース） | 遅い（サービス間通信・デプロイ基盤が必要） |
| デプロイの複雑さ | 低い | 非常に高い（Kubernetes 等の基盤が必要） |
| ローカル開発環境 | シンプル（Docker Compose 1つ） | 複雑（複数サービスの起動が必要） |
| ポートフォリオ用途 | 十分 | オーバーエンジニアリング |
| 将来のマイクロサービス化 | ドメイン境界を守れば移行可能 | 最初からマイクロサービスと同等 |

ポートフォリオプロジェクトとして、個人開発での**開発速度と学習効果**を最優先した。

### モジュラーモノリスの「モジュラー」を守る理由

単なる「スパゲッティモノリス」にならないよう、以下を徹底する:

1. **ドメイン間の直接 import 禁止** — `domain` パッケージ間の import を禁止し、interface または Spring Events 経由のみ許可
2. **集約ルート単位の Repository** — 他のドメインのデータには自身の Repository を通じてアクセスしない
3. **Bounded Context の明示** — パッケージ構造がマイクロサービスの境界に対応

これにより、将来マイクロサービス化する際のコスト（疎結合への変換）を最小化する。

---

## 結果

### 得られるもの

- 単一コードベースでの開発・テスト・デプロイ
- 同一トランザクション内でのクロスドメイン操作（複雑な分散トランザクションが不要）
- Render / Railway への単純なデプロイ

### 受け入れるトレードオフ

- ドメインごとのスケールアウトはできない（将来マイクロサービス化で解決）
- ドメイン間の境界を破るコードが混入しやすい（コードレビューで防ぐ）
- デプロイが全ドメイン一括になる

### 将来のマイクロサービス化パス

ドメインパッケージの境界を守ることで、将来必要になった際に以下の手順でマイクロサービスに移行できる:

1. 各ドメインパッケージを別 Spring Boot プロジェクトに分離
2. ドメイン間通信を HTTP API / Kafka に変換
3. データベースをドメインごとに分割
