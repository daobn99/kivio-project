# 提案依頼書（RFP）
# マルチベンダー型マーケットプレイス「Kivio」

**作成日：** 2026年5月23日
**作成者：** プロジェクトオーナー
**バージョン：** 1.0

## 1. プロジェクト概要

### 1.1 背景

個人や小規模事業者が、自分のショップを簡単に開設して商品を販売できるオンラインマーケットプレイスを構築したいと考えています。既存のECプラットフォーム（Amazon、楽天など）は手数料が高く、出店のハードルも高いため、もっと手軽に使える「小さなマーケットプレイス」を目指します。

### 1.2 サービス名

**Kivio**

### 1.3 コンセプト

> 「誰でも5分でお店が開ける、売り手と買い手がつながるマーケットプレイス」

### 1.4 ターゲットユーザー

- **売り手（セラー）：** ハンドメイド作家、副業で物販をしている個人、小規模な事業者
- **買い手（バイヤー）：** ユニークな商品を探している一般消費者
- **管理者（アドミン）：** プラットフォームを運営・管理する我々自身

## 2. 実現したいこと

### 2.1 買い手として

私は買い手として、以下のことができるようにしたいです。

- 会員登録・ログインができる（Googleアカウントでもログインしたい）
- 商品をカテゴリーやキーワードで探せる
- 商品の写真・説明・価格・レビューを見られる
- 気に入った商品をカートに入れて、まとめて購入できる
- クレジットカードで安全に決済できる
- 購入後、注文の状態（確認中→発送済み→配達完了）をリアルタイムで確認できる
- 気になることがあれば、購入前に売り手にチャットで質問できる
- 新着商品やお気に入りのショップの更新情報を通知で受け取れる

### 2.2 売り手として

私は売り手として、以下のことができるようにしたいです。

- 会員登録後、セラーとして申請できる
- 自分のショップページを作れる（ショップ名、紹介文、ロゴ画像）
- 商品を登録できる（商品名、説明文、価格、カテゴリー、写真を複数枚アップロード）
- 注文が入ったら通知を受け取り、注文管理画面で確認・対応できる
- 買い手からのチャットメッセージにリアルタイムで返信できる
- 自分のショップの売上データをダッシュボードで確認できる（今月の売上、注文数、人気商品など）
- 売上金の振込（出金）を申請できる

### 2.3 管理者として

管理者として、以下のことができるようにしたいです。

- セラーの申請を承認・却下できる
- 不適切な商品やショップを非表示・削除できる
- 全体の売上・ユーザー数・注文数などをダッシュボードで確認できる
- ユーザーからの問い合わせやトラブルに対応できる

## 3. 画面イメージ

具体的なデザインは開発者にお任せしますが、最低限以下の画面が必要だと考えています。

### 3.1 共通

- トップページ（おすすめ商品、カテゴリー一覧、検索バー）
- 会員登録 / ログイン画面
- ユーザープロフィール設定画面

### 3.2 買い手向け

- 商品一覧画面（検索・フィルター・ソート機能付き）
- 商品詳細画面（写真ギャラリー、商品情報、レビュー、セラー情報）
- ショッピングカート画面
- 決済画面（配送先入力 + クレジットカード決済）
- 注文履歴画面
- 注文詳細画面（ステータスのリアルタイム表示）
- チャット画面（セラーとの1対1チャット）

### 3.3 売り手向け

- セラーダッシュボード（売上サマリー、最近の注文、メッセージ通知）
- 商品管理画面（一覧・登録・編集・削除）
- 注文管理画面（ステータス変更：確認中→発送済み→完了）
- チャット画面（バイヤーからのメッセージ一覧）
- ショップ設定画面
- 売上・出金管理画面

### 3.4 管理者向け

- 管理ダッシュボード（全体統計）
- ユーザー管理画面
- セラー申請承認画面
- 商品モデレーション画面

## 4. 重要な要件

### 4.1 セキュリティ

- パスワードは安全に保管してください
- クレジットカード情報は当サービスでは保持しない（決済代行サービスに任せる）
- 本人以外のデータにはアクセスできないようにする
- セラーが管理者の画面を見れたり、買い手が他人の注文を見れたりしないこと

### 4.2 リアルタイム性

- チャットメッセージは送った瞬間に相手に届くこと
- 注文ステータスの変更は、ページを更新しなくても反映されること
- 新しい通知はリアルタイムで表示されること

### 4.3 決済

- テスト環境での決済で構いません（本番のお金は動かさなくてよい）
- ただし、本番移行を想定した設計にしてほしい
- Stripeなどの決済サービスの利用を想定

### 4.4 画像アップロード

- 商品画像は1商品につき最大5枚
- 画像は自動的にリサイズ・圧縮して、ページの表示速度を落とさないようにしてほしい
- ショップのロゴやプロフィール画像もアップロードできること

### 4.5 パフォーマンス

- 商品一覧ページはスムーズにスクロールできること
- 画像の読み込みが遅くてユーザーが待たされないこと

### 4.6 レスポンシブ対応

- スマートフォンでも快適に使えること（PC / タブレット / スマートフォン対応）

## 5. 優先順位

### Must（必須）

- 会員登録・ログイン（メール + Google）
- ロール管理（バイヤー / セラー / アドミン）
- 商品のCRUD + 画像アップロード
- 商品検索・フィルター
- カート + 決済（Stripe テスト）
- 注文管理（ステータス管理）
- リアルタイムチャット
- 通知機能（最低限、アプリ内通知）

### Should（できれば）

- セラーダッシュボード（売上グラフ）
- 商品レビュー・評価
- お気に入り（ウィッシュリスト）
- 管理者ダッシュボード
- メール通知

### Nice to have（余裕があれば）

- 商品のレコメンド（「こちらもおすすめ」）
- 出金管理
- SEO最適化

## 6. 技術に関する要望

技術的な詳細は開発者の判断に任せますが、以下の点はお願いしたいです。

- バックエンドは **Java / Spring Boot** で開発すること
- フロントエンドは **Next.js (TypeScript)** で開発すること
- インフラは**vscodeのdevcontainer**で開発すること
- ソースコードは **GitHub** で管理し、READMEに環境構築手順を記載すること
- テスト環境にデプロイし、URLでアクセスして動作確認できるようにすること
- デモ用のサンプルデータ（ダミーのショップ・商品・ユーザー）を用意すること
- **将来のマイクロサービス展開を考慮したモジュラー設計を採用すること**（ユーザー・商品・注文・チャットなど、ドメインごとに明確に境界を分けた設計にしてほしい）
